 根因一：恢复循环 Bug（Round 3 = -$11.20）

 现象

 Round 3（06:00–06:05）共 publish 107 次，replace=72，其中 06:03 单分钟内 62 次 publish，56 次
  cause=recovery。

 06:03 publishes by slot:
   YES_BUY:  29 次 recovery
   YES_SELL: 29 次 recovery
   NO_BUY:   2 次 policy
   NO_SELL:  2 次 policy

 正常 round 单分钟最多约 5-8 次。本 round 的 replace_per_min=14.55（Codex 门限 < 4），超出上限
  3.6 倍。

 根因

 成交后的 recovery 机制触发了"乒乓循环"：

 YES_BUY fill → recovery 重置 YES_BUY/YES_SELL
 → YES_SELL 以旧价格仍在书中被成交
 → 再次 recovery
 → 再次成交
 → ... (29 轮)

 这导致系统在 YES 侧快速来回 buy/sell，但每次 sell 都是以旧价格（低于当时
 mid）成交，形成"低卖高买"的系统性亏损。

 证据链

 - 06:00:31 YES_BUY@0.510，mid=0.564 → 正常
 - 06:00:38 系统 recovery 下单 Sell Yes@0.540
 - 06:00:58 该 Sell 被 cancel（reprice）
 - 06:01:18 系统重新下 Sell Yes@0.410（mid 此时约 0.36 → 合理）
 - 06:01:34 YES_SELL@0.410 成交，但mid 已回升至 0.475
 - 结果：bought at 0.510，sold at 0.410 → -$0.50/share × 5 = -$2.50

 这个模式在 06:03 重复了 29 次，累计贡献了 R3 的 $11.20 损失中的大部分。

 修复方向

 coordinator_order_io.rs 中 recovery 发布路径：
 - 限速：每 slot 每 N 秒最多触发一次 recovery
 - 或：recovery 时先检查 shadow_target 与当前 mid 的距离，若距离 > threshold 则拒绝
 recovery，等下一 tick 正常 policy 触发
 - 或：在 recovery 发布前校验"新报价与 trusted_mid 的偏差 ≤ glft_min_half_spread_ticks × 2"

 机制

 在 BTC 5m 这类高波动市场，价格在 300 秒内移动幅度远超 10 ticks。当市场单向移动时：

 1. 报价单在旧价格被 adversely selected（taker 知道价格要动）
 2. 新 recovery 订单再被 selected
 3. 形成"顺价 taker，逆价 maker"的持续亏损

 这对应 PLAN_Codex 归因树的 A 类（模型方向错）：
 - trusted_mid 滞后于真实后续价格
 - fill 后短时间出现不利移动
 - BTC 5m 市场 taker 整体具有方向信息优势

 但从现有日志已经可以推断：adversely selected fill 占比显著超过正常 maker 期望水平。

 2.3 locked_pnl 的计算方式

 每份 HTML 的 "Locked Profit" 字段 = 最坏情况下的 P&L：

 locked_pnl = min(YES_qty, NO_qty) × $1.00 - total_spent

 等价于：任何一侧赢，都至少赚这么多。

 - 当 locked_pnl > 0：无论 BTC 涨跌，绝对赚钱（guaranteed arbitrage）
 - 当 locked_pnl < 0：仓位没有锁定，若某侧输则亏钱

 2.4 0x8dxd 成功案例（session 1）深度解剖

 时间：15:50:05–15:53:07（3分2秒），BTC 强势上行，YES 价从 0.50 升至约 0.65

 关键路径：
 1. 15:50:05：买 101 NO @ 50¢（市场认为 NO 有 50% 概率胜）→ locked = -$50.50
 2. 15:50:09：买 6+101 YES @ 46-47¢（快速平衡 net_diff）→ net_diff 接近 0
 3. 随后 BTC 上行：YES 升至 60-75¢，NO 暴跌至 4-10¢
 4. 15:51:21–15:52:39：大量买入 NO @ 21-37¢（大幅压低 NO 均价）
 5. 15:53:05：集中买入 NO @ 4-5¢（NO 暴跌时买入）→ avg_NO 从 0.40 → 0.209
 6. 最终：YES=1345 @ 0.5976，NO=1106.94 @ 0.209，pair_cost = 0.8066
 7. locked_pnl = 1106.94 × $1.00 - $1031.78 = +$75.15（若 YES 赢则 +$313.22）

 边 (edge) 的来源：NO 从 50¢ 暴跌到 4¢（BTC 上行 → NO 被定价为几乎不可能赢）。但历史上 BTC 在
 5 分钟内方向反转的概率 >> 0%，所以 4¢ 的 NO 具有极高 EV（真实 EV 约 50¢ vs 买价
 4¢）。反复买入这种"极端便宜"的一侧，就能将 pair_cost 压低到远低于 1.0。

 关键路径：

 5. 结果：avg_YES = 0.70，avg_NO = 0.47 → pair_cost = 1.17-1.20
 6. 此时 pair_cost > 1.0：任何结局都亏钱，locked_pnl ≈ -$230

 失败根因：NO 从 0.50 下跌但幅度不够大（只到 0.47 avg），而 YES 却暴涨到 0.90+。要平衡
 net_diff 必须买昂贵的 YES，但买贵了就让 pair_cost 超过 1.0。策略的 "风险剧本"
 是：市场单边单调移动且没有回弹，使得无法低价买入两侧。

 2.7 策略边 (edge) 的完整逻辑链

 前提：Polymarket btc-updown-5m 市场每轮必定有一个赢家（YES=$1 或 NO=$1）

 边的本质：
   当市场价格偏离 0.50 时，偏低的一侧 EV >> 价格
   → 例：YES=0.05¢ 意味着市场认为 BTC 上行概率极低
   → 但实际上即使 BTC 走平，5 分钟后 YES 价格可能回升
   → 买入 0.05¢ 的 YES 的期望 >> 5¢

 成功路径：
   T=0s  市场均衡：YES=NO=0.50
   T=60s BTC 大涨：YES=0.70，NO=0.30
   T=120s BTC 继续涨：YES=0.85，NO=0.15
   T=180s BTC 暴跌：YES=0.30，NO=0.70
   → 整个过程 NO 曾经便宜（0.15），YES 也曾经便宜（0.30）
   → 分批买入两侧，实现 avg_YES ≈ 0.50，avg_NO ≈ 0.25 → pair_cost=0.75 → 大幅盈利

 失败路径：
   T=0s  YES=NO=0.50
   T=60s BTC 大涨：YES=0.70，NO=0.30  → 买了 NO（便宜）
   T=120s BTC 继续涨：YES=0.90，NO=0.10 → 买了 NO（更便宜）但现在要配对 YES 很贵
   T=300s 市场结算：YES 赢
   → avg_NO = 0.20（便宜），但平衡需要买 avg_YES = 0.75 → pair_cost=0.95 → 薄利
   → 若还有多余 NO：收 NO×0 = 0，损失 NO 均价 0.20/份

 GLFT 为什么在 BTC 5m 注定亏损：GLFT 的 edge 假设是"bid-ask spread > 被逆向选择的损失"。但在
 BTC 5m 高波动市场，taker 有信息优势（他们知道方向），systematically
 以过时价格成交，导致"高买低卖"。我们的 11 round 实盘已经证明这一点。

 Pair Arb 为什么可能有 edge：不需要比 taker 更快或更准确。只需要：(1) 等待极端价格出现，(2)
 在极端价格时大量买入，(3)
 等待结算。没有"被逆向选择"的问题，因为你是主动买方，不是挂单等对手方。

 Part 3：Pair Arb v2 重建方案

 3.1 设计目标

 仿照 0x8dxd / k9Q2mX4L8A7ZP3R 的核心策略：
 1. BUY-only maker（不再挂 SELL 单）
 2. 同时维护 YES 和 NO 两侧的买单
 3. 严格控制 |net_diff| ≤ max_net_diff（目标 2.0）
 4. 主要指标：pair_cost 和 locked_pnl（而非 replace_ratio）

 3.2 核心决策逻辑

 每个 tick（约 100-500ms 周期）：
   1. 读取 YES_ask, NO_ask（当前市场最优卖价）
   2. 计算 yes_eq = yes_avg_cost + NO_ask（如果买 NO 完成配对后的 pair_cost）
      计算 no_eq  = NO_avg_cost + YES_ask（如果买 YES 完成配对后的 pair_cost）
   3. 决策：
      a. 如果 YES_ask < buy_threshold（默认 0.45）且 net_diff >= -max_net_diff → 发 BUY YES
      b. 如果 NO_ask < buy_threshold（默认 0.45）且 net_diff <= +max_net_diff → 发 BUY NO
      c. 若 |net_diff| >= max_net_diff：强制买入落后侧（不管价格高低）
      d. 若 pair_cost + min(YES_ask, NO_ask) >= pair_cost_ceiling（1.05）→ 暂停买入
   4. 订单大小（qty）：
      - 基础 qty = PM_BID_SIZE（5 份）
      - 若价格 < cheap_threshold（0.25）：qty × price_inverse_factor（更多）
      - 若 |net_diff| 接近 max_net_diff：强制 qty = max(1, max_net_diff - |net_diff|)
